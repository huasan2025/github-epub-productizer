# Gemini API 生图使用规范

这份规范不是为了把 Gemini 管得很死，而是为了避免把简单事情做复杂。

核心原则只有一句：

**充分发挥 Gemini 自己的 AI 能力，只给它准确的任务描述和准确的上下文参考图。**

不要过度控制，不要写一大堆机械限制，不要用复杂脚本替代模型本来就擅长做的事。

---

## 1. 什么时候适合直接用 Gemini 生图

适合 Gemini 的场景：

- 有明确视觉目标，但不需要像素级精确控制
- 有参考图，可以告诉它“结构参考谁、内容参考谁、主视觉参考谁”
- 想生成一张完整海报、商品图、插画、封面图、概念图
- 更看重整体感、营销感、构图感，而不是前端级别的严格排版

不适合 Gemini 独立完成的场景：

- 文字必须 100% 精准、字号位置必须稳定
- 需要严格品牌规范、严格栅格、严格对齐
- 需要输出可反复编辑的结构化设计稿

这类场景优先用：

- HTML/CSS -> PNG
- Figma
- 或“Gemini 出视觉底稿 + HTML/设计工具做最终定稿”

---

## 2. 最重要的原则：少写要求，多给参考

Gemini 生图时，最有效的输入通常不是超长 prompt，而是：

1. **一句清楚的任务描述**
2. **1 到 3 张高质量参考图**
3. **一句说明每张参考图分别负责什么**

坏例子：

- 写 500 字风格要求
- 限制一堆颜色、字体、构图、情绪、镜头语言
- 用很多“不要这样，不要那样”去绑模型

好例子：

- 给封面图制作一张营销海报
- 结构参考图 1
- 文案层级参考图 2
- 主视觉使用图 3

Gemini 很擅长自己补全中间过程。  
我们真正要做的，是把**目标**和**上下文**喂准。

---

## 3. 推荐输入结构

每次做图，优先按这个顺序准备：

### A. 任务句

用一句话说明你要什么：

- 给这本书制作一张营销海报
- 给这本电子书生成一张闲鱼商品封面
- 给这篇文章做一张概念插画
- 给这个产品做一张带封面和卖点的商品图

### B. 参考图角色说明

如果有多张图，一定要告诉 Gemini 每张图是干什么的：

- 图 1：结构参考
- 图 2：文案层级参考
- 图 3：主视觉 / 封面图 / 产品图

这一句非常重要。  
不是“我给你三张图，你自己看”，而是明确告诉它：

**哪张图决定结构，哪张图决定内容感觉，哪张图决定主体。**

### C. 保留少量关键约束

只保留最关键的几个要求，比如：

- 风格偏闲鱼商品封面
- 字要大，缩略图里也能看清
- 适合中文商品页
- 重点突出封面图，不要做成纯背景海报

到这里就够了。  
不要继续把 prompt 写成产品需求文档。

---

## 4. 推荐工作流

### 工作流 A：多参考图直接生成

最推荐，尤其适合商品图、营销图、海报。

输入：

- 任务描述
- 结构参考图
- 文案参考图
- 主视觉图 / 封面图

输出：

- 一张完整成图

适合场景：

- 闲鱼商品封面
- 海报
- 资料包展示图
- 电商转化图

### 工作流 B：先 Gemini，再人工挑选

适合需要创意感的时候。

做法：

- 同一套输入连续生成 3 到 5 张
- 不要试图一次命中
- 从里面挑结构最对的一张继续迭代

Gemini 生图不是“精确求解”，而更像“方向正确后抽样”。

### 工作流 C：Gemini + HTML 混合

适合“既要营销感，又要文字百分百可读”。

做法：

- Gemini 负责背景、氛围、商品主视觉
- HTML/CSS 负责最终文字和按钮排版

适合场景：

- 商品图里的小字很多
- 需要一键修改文案
- 需要稳定复用同一个模板

---

## 5. Prompt 写法建议

### 写法 1：一句话任务 + 参考图分工

最常用。

示例：

> 给封面图制作一张营销海报，结构参考图 1，内容层级参考图 2，主视觉使用图 3，整体风格偏闲鱼商品封面，字要大，重点突出这是一本 Google 软件工程经典电子书。

### 写法 2：一句目标 + 一句重点

适合已经有参考图的时候。

示例：

> 做一张适合中文商品页的电子书营销图。重点是让人一眼看懂这是什么书、适合谁、为什么值得买。

### 写法 3：只补最关键的视觉要求

比如只说：

- 字大
- 缩略图清楚
- 商品感强
- 不要做成纯艺术海报

就够了。

---

## 6. 不建议做的事

### 不要把 prompt 写得像程序

比如：

- 顶部标题字号 80
- 左边 40%
- 右边 60%
- badge 三个
- 每个模块 32 像素 padding

这些适合 HTML，不适合 Gemini。

### 不要给太多风格词

比如一口气塞：

- 极简
- 科技
- 高级
- 电商
- 未来感
- 信息密集
- 品牌感
- 高转化
- 大气
- 视觉冲击

这种词太多，模型反而抓不住重点。

### 不要用复杂脚本替代多模态输入

如果目标是一张完整营销海报，优先让 Gemini 一次看懂：

- 任务
- 参考结构
- 参考内容
- 主视觉

不要先让 Gemini 只生背景，再用 Python 拼一堆东西，除非你明确需要严格可控。

---

## 7. 什么时候优先用参考图，而不是补文字

只要你脑子里已经有“像什么”的感觉，就优先找参考图。

因为对 Gemini 来说：

- 一张准确的参考图，通常比 100 字描述更有效

优先级通常是：

1. 参考图是否准确
2. 任务描述是否清楚
3. 关键约束是否少而准
4. 文字 prompt 是否优美

最后一项其实最不重要。

---

## 8. Gemini API Key 位置

Gemini API key 可以在本机的：

- `~/.zshrc`

里找到。

调用脚本时，优先通过环境变量使用：

- `GEMINI_API_KEY`

不要把 key 硬编码进脚本、文档或仓库文件里。

---

## 9. 推荐默认策略

以后只要调用 Gemini API 生图，默认先按这个顺序思考：

1. 这张图是更适合 Gemini，还是更适合 HTML？
2. 如果适合 Gemini，我有没有 1 到 3 张准确参考图？
3. 我能不能把任务描述压缩成一句清楚的话？
4. 我是不是写了太多没必要的限制？

如果答案是：

- 有参考图
- 目标明确
- 不要求像素级控制

那就直接让 Gemini 发挥，不要过度干预。

---

## 10. 一句话版本

**Gemini 生图最有效的方式，不是写更复杂的 prompt，而是提供更准确的任务描述和更准确的参考图分工。**
