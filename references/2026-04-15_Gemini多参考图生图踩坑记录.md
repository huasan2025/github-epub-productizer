# Gemini 多参考图生图踩坑记录

这次最值钱的经验，不是“又做出了一张图”，而是确认了：

**Gemini API 不是不行，问题往往出在调用方式。**

---

## 结论先说

如果目标是做一张完整的商品营销图，默认优先级应该是：

1. **Gemini 多参考图直接生成**
2. **HTML/CSS -> PNG** 作为稳定 fallback
3. 不要默认走“Gemini 只生背景 + Python 再拼图”的路线

原因很简单：

- 多参考图方式最容易得到完整海报感
- HTML 方式最容易保证文字可读和排版稳定
- “只生背景再拼装”两头不占，既没有整图感，也不够稳定

---

## 这次真正的 bug 是什么

前面几轮效果差，不是因为 Gemini 模型差，也不是因为 prompt 不够长。

真正的问题有 3 个：

### 1. 把多模态问题做成了纯文本问题

手动在 Gemini 官网成功的那次，输入其实是：

- 图 1：结构参考
- 图 2：内容层级参考
- 图 3：干净封面图
- 外加一句清楚任务描述

这是一个标准的**多参考图生成任务**。

而前面 API 版本很多次走的是：

- 只给文本 prompt
- 或只让 Gemini 生成背景
- 再由脚本去拼封面和文字

这等于把 Gemini 最擅长的部分绕开了。

### 2. 过度相信脚本控制，低估了模型整体构图能力

当目标是：

- 一张像闲鱼商品封面的营销海报
- 既有商品感，又有标题、卖点、视觉焦点

这类任务其实更适合让 Gemini 一次看懂整体。

脚本更适合：

- 精准排版
- 固定模板
- 小字必须清楚

不适合替代模型去拼“海报整体感”。

### 3. 参考图分工没有明确喂给模型

最有效的不是“上传三张图”，而是明确告诉模型：

- 哪张图负责结构
- 哪张图负责内容层级
- 哪张图负责主视觉

没有这句话，模型只能自己猜。

一旦猜错，就会出现：

- 把不该保留的年份也继承过去
- 把参考图里的旧文案沿用
- 结构像了，但主体不突出

---

## 正确做法

以后只要是商品图、营销图、海报，默认优先尝试：

### 方案 A：Gemini 多参考图

输入尽量简化成这 3 部分：

1. 一句任务描述
2. 1 到 3 张参考图
3. 一句说明每张图分别负责什么

示例结构：

> 给这本书制作一张中文商品页营销海报。  
> 图 1 是结构参考，参考它的商品封面感和信息密度。  
> 图 2 是内容层级参考，参考它的标题、卖点和底部信息组织方式。  
> 图 3 是主视觉封面图，这本书必须作为海报主体明显展示。  
> 整体风格偏闲鱼商品封面，字要大，缩略图里也能看清。

重点：

- 少写规则
- 少写程序式布局要求
- 多给准确参考

### 方案 B：HTML/CSS

当满足以下任一条件时，改用 HTML：

- 文字必须 100% 可读
- 小字很多
- 需要反复改文案
- 需要一键导出 PNG
- 需要稳定复用同一个版式

HTML 适合做“稳定模板”，不适合替代 Gemini 做“整图创意生成”。

---

## 模型经验

这次实际测试下来：

- `gemini-3.1-flash-image-preview` 可以直接完成多参考图营销海报生成
- `gemini-3-pro-image-preview` 可作为 fallback
- 不要默认退回到 `gemini-2.5-flash-image`

推荐顺序：

1. `gemini-3.1-flash-image-preview`
2. `gemini-3-pro-image-preview`
3. 两者都不行时，把 prompt 和参考图组合导出，改走 Gemini 官网手动生成

---

## 这次确认下来的默认策略

以后做 Gemini 商品图时，默认按这个策略：

### 默认策略

- 先判断是不是“整图营销海报”
- 如果是，优先走 Gemini 多参考图
- 如果失败，再走 HTML 模板

### 不再默认采用

- Gemini 只生背景
- Python 把封面、文字、背景硬拼到一起

这条路可以作为临时实验，不再作为默认方案。

---

## 一句话沉淀

**Gemini 生图效果差，很多时候不是模型不行，而是我们把应该交给多模态模型做的事，错误地拆成了“文本 prompt + 脚本拼装”。**
